# 840D sl OPC UA `too many monitored items` 问题分析与根治方案

## 1. 文档目的

本文用于系统说明西门子 SINUMERIK 840D sl 在 OPC UA 订阅场景下出现 `The request could not be processed because there are too many monitored items in the subscription.` 的原因、影响因素、最大负载计算规则，以及如何从底层彻底解决问题，并尽量避免对数控系统稳定性造成影响。

## 2. 问题现象

在 840D sl 的 OPC UA 客户端接入过程中，当客户端为大量点位创建订阅（Subscription）与监控项（MonitoredItem）时，可能出现以下错误：

- `The request could not be processed because there are too many monitored items in the subscription.`

该错误说明：当前 OPC UA Server 无法继续承受新增监控项，或者当前订阅设计已经超出了 Server 的资源边界。

## 3. 产生问题的根本原因

### 3.1 本质原因：监控项负载发生在 840D sl 内部

840D sl 自带的 OPC UA Server 运行在数控系统内部。客户端每创建一个 MonitoredItem，Server 都需要在控制器侧持续执行以下工作：

1. 保存监控项配置。
2. 按采样周期检查目标节点。
3. 判断数据是否变化。
4. 维护数据变更队列。
5. 按发布周期组包并发送给客户端。
6. 维护 Session、Subscription、KeepAlive、重传等状态。

因此，订阅压力不是在客户端产生，而是在 840D sl 内部产生。后端再增加 MQTT、二级 OPC UA Server 或数据库，只能缓解 840D sl 后面的分发压力，不能消除 840D sl 这一段已经发生的监控项负载。

### 3.2 根本矛盾：控制器不是为大规模数据中台设计的

840D sl 的首要任务是数控插补、运动控制、工艺执行和 PLC/NC 实时任务。其 OPC UA Server 是标准通信接口，不是为无限扩展的数据分发平台设计的，因此天然存在会话数、订阅数、监控项处理能力和离散采样周期等边界。

## 4. 导致问题的主要因素

### 4.1 监控项总数过多

只要节点最终属于 840D sl 的 OPC UA 地址空间，并且客户端对这些节点创建了 MonitoredItem，那么无论这些点位在业务上属于 CNC、PLC、GUD、参数/设定值、AGV 映射数据还是其他外挂业务对象，都会共同占用 840D sl 的订阅负载预算。

### 4.2 采样周期过短

采样周期越短，Server 每秒需要执行的检查次数越多。

- 100 ms = 每秒 10 次
- 250 ms = 每秒 4 次
- 500 ms = 每秒 2 次
- 1000 ms = 每秒 1 次

真正决定 OPC UA 监控负载的核心参数，是采样周期，不是发布周期。

### 4.3 多个客户端重复订阅

若现场同时存在网关、SCADA、MES、看板系统、UaExpert 或自研程序等多个客户端直连 840D sl，则每个客户端都会创建自己的 Session、Subscription 和 MonitoredItem，导致总负载叠加。

### 4.4 外部系统数据挂在 840D sl 地址空间中

如果 AGV、机器人、立库等外部系统的数据被映射到了 840D sl 的 OPC UA 地址空间里，那么这些数据一旦被订阅，也同样会消耗 840D sl 的监控项资源，挤占本该留给 CNC 核心实时数据的预算。

### 4.5 订阅设计过于原子化

如果为了得到“机床运行状态”“加工开始/结束”“业务看板数据”，直接订阅大量底层原始点，那么客户端虽然拿到了细颗粒度数据，但 Server 端的监控项数量和采样负载会快速膨胀。

### 4.6 客户端重连或配置错误导致重复创建订阅

若客户端重连后没有正确释放旧订阅，或者配置中重复订阅同一个节点，也会导致 Server 侧出现“逻辑上 100 个点，实际资源占用远大于 100 个点”的情况。

## 5. 840D sl 中常见的数据类型

840D sl 的 OPC UA 地址空间中，通常不仅包括 CNC/NC 实时数据，还可能包括：

- CNC / NC 数据
- PLC 数据
- GUD（Global User Data）
- Machine Data
- Setting Data
- 报警/事件对象
- 文件系统/刀具管理等方法对象

这些数据类型在业务含义上不同，但只要最终以 MonitoredItem 的方式挂在 840D sl 上被监视，就会共同消耗同一个订阅负载预算。

### 5.1 特别说明：第三方设备主动写入 GUD 时，是否还存在“采样周期”问题

如果外接设备不是“客户端去读或订阅外部设备”，而是**第三方主动通过 OPC UA Write 将数据写入 840D sl 的 GUD**，那么要把两个过程分开看：

#### 过程 A：第三方设备写入 GUD

这一段本质上是 **Write 操作**，不是 MonitoredItem 采样。

- 写入频率高低会影响 840D sl 的通信与变量更新压力；
- 但“写入动作本身”不按 MonitoredItem 的采样周期公式计算；
- 因此，**外部设备主动写入 GUD，本身没有“采样周期”这一说**。

#### 过程 B：其他客户端再去订阅这些 GUD 节点

一旦其他客户端对这些 GUD 节点创建了订阅，840D sl 仍然要对这些 GUD 节点建立 MonitoredItem，并按采样周期检查与发布变化。

也就是说：

- **写入端**关注的是写入频率、写入并发、写入节点数量；
- **订阅端**关注的才是采样周期、发布周期、MonitoredItem 数量与总 items/s。

因此，如果第三方设备高频写入 GUD，而上层系统又高频订阅这些 GUD，那么 840D sl 会同时承受：

1. GUD 被频繁写入带来的变量更新压力；
2. GUD 被订阅带来的 MonitoredItem 采样与发布压力。

这两类压力会叠加，但它们不是同一种负载。

## 6. 最大负载的计算规则

### 6.1 计算原则

840D sl 的订阅能力评估，不是只看点位个数，而是要看：

- 所有活动会话（Sessions）
- 所有订阅（Subscriptions）
- 所有监控项（MonitoredItems）
- 各自的采样周期（Sampling Interval）

共同形成的总采样负载。

### 6.2 实用计算公式

```text
总负载（items/s）≈ Σ [每个监控项每秒采样次数]
```

也就是：

```text
总负载（items/s）≈ Σ [1 / 采样周期(秒)]
```

如果同一类点位数量相同，则可写为：

```text
某类点位负载 = 点位数量 × (1 / 采样周期(秒))
```

所有类别相加后，得到总负载。

### 6.3 适用边界：该公式只适用于“订阅负载”，不直接适用于“写入负载”

上面的 `items/s` 公式用于估算 **Subscription / MonitoredItem** 造成的订阅负载。

如果当前场景是：

- 第三方设备主动写入 840D sl 的 GUD；
- 然后其他客户端再去订阅这些 GUD；

那么要拆成两部分理解：

#### 写入负载

- 与写入频率有关；
- 与每次写入的节点数量有关；
- 与是否多个客户端并发写入有关；
- 不直接按“采样周期”计算。

#### 订阅负载

- 与订阅这些 GUD 的 MonitoredItem 数量有关；
- 与采样周期有关；
- 与发布周期和订阅组织方式有关；
- 可以用本文的 `items/s` 公式做估算。

因此，**“第三方高频写入 GUD”不会自动等价成“高采样周期”，但它会让被订阅的 GUD 更频繁发生变化，从而放大发布与处理压力。**

### 6.4 典型示例

#### 示例 1：全部 100 ms 采样

- 100 个点 × 10 次/秒 = 1000 items/s

说明：已经接近或达到上限。

#### 示例 2：混合采样周期

- 20 个点，100 ms：20 × 10 = 200
- 30 个点，250 ms：30 × 4 = 120
- 40 个点，1000 ms：40 × 1 = 40
- 50 个点，5000 ms：50 × 0.2 = 10

总负载：

```text
200 + 120 + 40 + 10 = 370 items/s
```

说明：这种结构明显比“全部高频订阅”更安全。

#### 示例 3：第三方设备高频写入 GUD，但订阅端采样较低

假设：

- 第三方每 100 ms 向 20 个 GUD 变量写入一次；
- 上层客户端订阅这 20 个 GUD，但采样周期配置为 1000 ms；

则要分开理解：

- **写入侧**：840D sl 每秒要处理 20 × 10 = 200 次变量写入更新；
- **订阅侧**：20 个 MonitoredItem × 1 次/秒 = 20 items/s；

这说明：

- 写入很频繁；
- 但订阅负载并没有按 200 items/s 计算，而仍按采样周期算 20 items/s；
- 不过由于底层值变化频繁，发布队列和变化通知压力可能仍会上升。

### 6.5 发布周期的作用

发布周期（Publishing Interval）决定的是 Server 多久将已经采集到的数据通知发给客户端一次。它主要影响网络发送频率、客户端接收频率和 Publish 打包行为，但不直接决定 Server 内部采样的检查频率。因此，若只增大发布周期而不增大采样周期，往往只能减少网络与客户端压力，不能从根本上降低 840D sl 的监控项采样负载。

### 6.6 轮询 + 订阅数量一旦都很多，是否会导致服务崩掉

会，**虽然轮询不会触发 `too many monitored items in the subscription` 这种订阅专属报错，但“高频轮询 + 大量订阅”叠加后，仍然可能把 840D sl 的 OPC UA 服务推到不稳定状态。**

要特别区分两件事：

- **订阅过载**：更容易表现为 MonitoredItem 超限、Subscription 创建失败、通知队列积压；
- **总体服务过载**：更容易表现为响应变慢、Read 超时、会话掉线、OPC UA 服务卡顿，严重时甚至让 OPC UA 通信看起来像“崩掉”。 

也就是说，轮询只是绕开了“MonitoredItem 数量限制”这一类错误，**并没有绕开 OPC UA Server 的整体 CPU、内存、线程、通信处理能力边界。**

#### 可以把总压力理解成三部分叠加

1. **写入压力**：第三方高频写入 GUD 或其他变量；
2. **订阅压力**：大量 MonitoredItem 的采样、变化判断、队列维护、发布；
3. **轮询压力**：客户端高频 Read，大批量读节点，多个客户端并发读。

当这三类压力叠加后，即使订阅点数没有继续增长，也可能出现：

- OPC UA Read 响应明显变慢；
- KeepAlive 延迟；
- Session 断开；
- 数据通知延迟明显增大；
- 客户端误以为 Server “崩掉”或“不稳定”。

#### 一个简化判断原则

- **订阅多**：容易先报 `too many monitored items` 或创建失败；
- **轮询猛**：容易先表现为读超时、卡顿、响应慢；
- **两者同时多**：最容易出现整体服务不稳定。

因此，工程上不能把“改成轮询”理解成“系统就一定安全”，而应理解成：

> **轮询只是把问题从“订阅对象过多”转移成“总体读负载过高”的风险。**

## 7. 为什么中间件不能单独解决根因

在 Windows 或 IPC 上增加 MQTT Broker、二级 OPC UA Server、历史库或边缘代理，能够解决的是多个上层系统重复访问 840D sl、数据分发能力不足、上层系统协议不统一和并发订阅压力大等问题。

但如果 840D sl → 边缘采集器 这一段，仍然是 100+ 原始点、高频采样和大量 MonitoredItem，那么问题依然发生在 840D sl 内部。后面的中间件只是接力分发，无法替 840D sl 承担已经发生的采样、变化检测和订阅队列维护负担。

如果外部设备采用的是“主动写入 GUD”的模式，那么中间件同样不能消除 840D sl 这一端的写入处理压力；它能缓解的依然只是 840D sl 之后的分发压力。因此在 GUD 写入场景下，也应继续遵循“少量高价值点进入 840D sl，复杂聚合放到边缘层”的原则。

## 8. 彻底解决问题的总体思路

要从底层彻底改变问题，并尽量不影响数控系统稳定性，必须遵循两个原则：

1. 减少 840D sl 需要高频监控的对象数量。
2. 让 840D sl 只面对一个受控采集器，而不是多个上层系统。

## 9. 根治方案一：做数据语义压缩

### 9.1 状态字压缩

将多个布尔状态压缩为一个整型状态字，例如自动模式、手动模式、运行中、暂停、报警、急停、门状态和主轴状态，可压缩为 `MachineStatusWord`，从而将多个监控项减少为 1 个监控项。

### 9.2 业务状态枚举

将多个底层点计算为一个上层业务状态，例如 `IDLE`、`READY`、`RUNNING`、`HOLD`、`ALARM`、`TOOL_CHANGE`，使上层系统不再需要订阅十几个底层位去自行拼装状态。

### 9.3 事件化输出

将高频连续监视改为关键事件输出，例如：

- 加工开始
- 加工结束
- 报警产生
- 报警恢复
- 程序切换
- 刀具更换

事件化可显著减少对持续高频订阅的依赖。

### 9.4 快照对象

对于低频关心的数据，生成快照对象，例如当前程序号、当前工件号、当前刀具号、今日产量、班次产量和最后报警号，通过快照轮询而非逐点高频订阅来获取。

## 10. 根治方案二：重构采集分层

### 10.1 仅保留少量核心高实时订阅

只对真正有高实时价值的点位做订阅，例如运行/停机、报警状态、急停、程序切换、模式切换和节拍关键位。

### 10.2 可推导数据不直接高频订阅

如“机床是否加工中”“当前工艺阶段”“设备是否可加工”等，改为在设备侧或边缘侧由少量输入点推导得出。

### 10.3 低价值或低频数据改为轮询

例如统计值、设定值、参数、快照数据和外围系统状态，这些数据没有必要长期占用高频订阅资源。

但这里要注意：**轮询不是没有代价。** 如果轮询点位数量很多、读周期很短、同时还有大量订阅存在，那么系统风险只是从“监控项超限”变成“总体 Read 负载过高”。因此低价值数据改轮询时，必须同时控制：

- 单次读取的点位数量；
- 轮询周期；
- 并发读取客户端数量；
- 是否与高频订阅同时存在。

### 10.4 GUD 写入场景下的额外原则

如果外接设备是主动写入 GUD，而不是由 840D sl 去采集外设，那么还应补充以下控制原则：

- 不要把大量外设原始点逐个高频写入 GUD；
- 优先写入状态字、事件码、业务枚举、快照对象；
- 对高频变化量优先在外设侧或边缘侧先聚合，再写入少量结果变量；
- 避免把 840D sl 当成多外设实时数据汇聚总线。

### 10.5 ThingsBoard Gateway “分批订阅”是否能解决问题

可以缓解，但**通常不能单独根治**。

如果在 ThingsBoard Gateway 里把点位拆成多个批次（多个 Subscription 或多个配置组）：

- **能改善的**：
  - 避免单个 Subscription 一次挂载过多 MonitoredItem；
  - 便于按业务给不同批次设置不同采样周期；
  - 便于定位哪一批次导致超限或不稳定。

- **不能自动改善的**：
  - 840D sl 侧“总 MonitoredItem 预算”；
  - 840D sl 侧“总 samples/s 预算”；
  - 高频轮询 + 高频订阅 + 高频写入叠加造成的总体服务负载。

因此要明确：

> **分批订阅优化的是“组织方式”，不是“总负载上限”。**

如果总负载仍然超过 840D sl 可承受边界，分批后仍可能出现：

- 监控项相关报错（例如某批次创建失败）；
- Read 响应变慢、会话抖动、通知延迟增大等总体不稳定表现。

#### 推荐落地方式（ThingsBoard Gateway）

1. 先按业务价值分批：核心实时、普通状态、低频统计。
2. 每个批次分别设置采样周期与发布周期，不要统一高频。
3. 对每批次做负载预算（items/s 估算），并统计总和。
4. 若总和仍高，优先做点位压缩（状态字、事件化、快照化），而不是继续“只拆批次不减总量”。
5. 长期仍建议采用“单边缘采集器 + 上层分发”架构，避免多个系统重复消耗 840D sl。

## 11. 根治方案三：引入单一边缘采集器

### 11.1 架构原则

只允许一个边缘采集器连接 840D sl，所有其他系统都从边缘层取数据。

### 11.2 推荐架构

```text
[840D sl OPC UA Server]
        ↓
[边缘采集代理 / IPC / Windows]
  ├─ OPC UA Client
  ├─ 状态字合并
  ├─ 业务状态机
  ├─ 事件提取
  ├─ 快照聚合
  ├─ 缓存 / 去重 / 质量码处理
  └─ 断线重连保护
        ↓
  ├─ MQTT
  ├─ 二级 OPC UA Server
  ├─ API / DB
        ↓
[MES / SCADA / 看板 / 报警平台 / 云平台]
```

### 11.3 作用

该方案能够避免多个系统直接访问 840D sl，避免重复创建 Session / Subscription，将复杂逻辑转移到高配置边缘设备上，并让上层系统复用同一份数据。

## 12. 根治方案四：将外围系统数据迁出 840D sl

如果 AGV、机器人、立库等数据当前挂在 840D sl 的 OPC UA 地址空间中，应尽量迁移到各自独立的 OPC UA Server、边缘平台、MQTT Broker 或统一数据中台，以避免挤占 CNC 核心实时数据预算，并改善故障隔离性与系统扩展性。

## 13. 根治方案五：建立边缘状态机与统一数据产品

边缘层不应只做“转发器”，而应输出统一、可复用的数据产品，例如：

- `MachineState`
- `AlarmState`
- `ProgramInfo`
- `CycleStartEvent`
- `CycleEndEvent`
- `ProductionSnapshot`

这样以后业务系统新增需求时，主要修改边缘层逻辑，而不是继续增加 840D sl 的监控项。

## 14. 实施顺序建议

### 第一步：止血

- 清点所有客户端连接
- 排查重复订阅
- 排查无效点位
- 暂时降低非关键点采样频率
- 暂时移除外围系统高频订阅
- 暂时拉长高频轮询周期，避免“轮询 + 订阅”共同把 OPC UA 服务压垮

### 第二步：减载

- 将点位按“核心实时 / 可推导 / 低频统计”分层
- 做状态字压缩
- 做业务状态枚举
- 做快照对象
- 将外围系统数据从 840D sl 迁出

### 第三步：重构

- 部署单一边缘采集器
- 建立边缘状态机
- 输出 MQTT / 二级 OPC UA / API
- 上层系统全部改连边缘层

## 15. 最终结论

`too many monitored items` 的根因，不是客户端软件本身太弱，而是：

- 840D sl 内置 OPC UA Server 能力有固定边界
- 监控项维护成本发生在 840D sl 内部
- 大量原子点高频订阅会快速耗尽其订阅预算
- 多系统直连和外围系统数据挂载会进一步放大该问题

因此，彻底解决问题的正确方向不是继续增加 840D sl 的订阅承载，而是：

1. 减少 840D sl 需要维护的高频监控项数量；
2. 将原始点压缩成状态字、业务状态、事件和快照；
3. 将复杂逻辑和多系统分发迁移到边缘层；
4. 让 840D sl 只面对一个受控采集器；
5. 将 AGV 等外围系统数据迁出 840D sl。

只有这样，才能从底层真正改变问题，并尽量避免对数控系统本体造成影响。
